# Contact API - Backend для лендинг-презентации разработчика

REST API для формы обратной связи с AI-интеграцией, email-уведомлениями и защитой от спама.

## Стек технологий

- **Backend:** PHP 8.4+, Symfony 8.1
- **База данных:** SQLite (через Doctrine DBAL)
- **AI:** OpenAI-совместимое API (LM Studio / Groq / OpenAI)
- **Email:** SMTP (Mailtrap для тестирования)
- **Документация:** статический OpenAPI + Swagger UI (`public/docs/`)
- **Логирование:** Monolog (в файлы)

## Архитектура

Слоистая структура: **Controllers → Services → Repositories**

**Паттерны:** Dependency Injection, Repository, DTO.

### Объяснение выбора технологий

**PHP + Symfony** - для небольшого REST API нужен был фреймворк с готовой инфраструктурой: маршрутизация, DI-контейнер, валидация, HTTP-клиент, почта, логирование. В Symfony всё это есть из коробки и хорошо стыкуется между собой, поэтому не пришлось собирать стек из отдельных библиотек.

**SQLite + Doctrine DBAL** - проект небольшой: две таблицы, без сложных связей. SQLite не требует отдельного сервера БД, база - один файл (`var/data.db`), что упрощает установку и деплой.

**OpenAI-совместимое API** - формат `/v1/chat/completions` поддерживают LM Studio, Groq и OpenAI. Провайдер меняется через `.env`, код остаётся тем же. AI вынесен во внешний сервис: бэкенд только отправляет запрос и парсит JSON-ответ.

**Symfony Mailer + SMTP** - стандартный способ отправки писем в экосистеме Symfony. Mailtrap удобен для локальной отладки без реальной отправки.

**Monolog** - стандарт логгирования в Symfony. Разделение на `api_requests.log` и `app.log` позволяет отдельно смотреть трафик и ошибки.

**Rate limiting через SQLite** - счётчик запросов переживает перезапуск сервера, работает быстро.

## API

| Метод |      Путь      |           Описание          |
|-------|----------------|-----------------------------|
| POST  | `/api/contact` | Отправить обращение         |
| GET   | `/api/health`  | Проверка состояния (БД, AI) |
| GET   | `/api/metrics` | Статистика обращений        |

Все запросы должны содержать заголовок `Accept: application/json` - иначе Symfony может вернуть HTML вместо json.

Документация: http://localhost:8000/docs/ (Swagger UI, статическая спека в `public/docs/openapi.yaml`).

### AI-интеграция

Сервис отправляет текст обращения во внешнее OpenAI-совместимое API и ожидает JSON с тремя полями:

- **sentiment** - тональность: позитивно / нейтрально / негативно
- **type** - категория: отзыв / сотрудничество / вопрос / жалоба / другое
- **reply** - автоответ ИИ-помощника от имени разработчика

**Провайдер для тестирования:** LM Studio, модель `google/gemma-4-e4b` - хорошо справляется с классификацией и генерацией ответов.

**System prompt** (формируется в `AIService`):

```text
Ты - часть системы обратной связи лендинг-презентации разработчика.
Твоя задача - помогать сервису выполнять свои задачи.
Отвечай ТОЛЬКО чистым JSON, без markdown-форматирования, без обрамляющих ```.

Поля:
  sentiment - тон сообщения, одно слово из списка: нейтрально, позитивно, негативно
  type      - категория обращения, одно слово из списка: сотрудничество, вопрос, жалоба, отзыв, другое
  reply     - ответ ИИ-помощника разработчика: поздороваться по имени, представиться,
              поблагодарить за обращение, сообщить что разработчик скоро ответит лично,
              кратко ответить по сути комментария
```

**User prompt:** `Посетитель сайта {name} оставил сообщение: {comment}`

### Fallback

Если AI недоступен - `sentiment` и `type` возвращают `null`, сервис продолжает работать. Поле `reply` всегда заполнено: AI-ответ или стандартная заглушка.

### Rate Limiting

Реализован через SQLite. Каждый запрос сохраняется с IP и временной меткой. При проверке:
- удаляются записи старше таймаута (60 секунд по умолчанию)
- подсчитываются свежие запросы от этого IP
- если больше лимита (5 по умолчанию) - возвращается 429

Настройки в `.env`: `RATELIMIT_LIMIT_REQUESTS`, `RATELIMIT_TIMEOUT_SECONDS`.

### Логирование

- `var/log/api_requests.log` - все запросы к API (метод, путь, IP, тело, статус)
- `var/log/app.log` - ошибки и предупреждения

### Обработка ошибок

Глобальный `ExceptionsListener` перехватывает все исключения и возвращает JSON вместо HTML. Для HTTP-исключений сохраняется оригинальный статус-код. В production-режиме (`APP_DEBUG=0`) детали ошибок скрыты.

### Хранение данных

- SQLite (`var/data.db`): таблицы `RateLimit` (IP и время запросов), `Feedback` (обращения)
- Логи: текстовые файлы в `var/log/`
- Миграции: SQL-файлы в `migrations/`
- **Статистика**: запрашивается из БД на лету через `GET /api/metrics`, отдельно не хранится

## Что сделано с помощью AI

Инструмент: **DeepSeek**. Symfony до этого не использовал - после NestJS общие идеи (контроллеры, DI, DTO, сервисы) давались легко, но синтаксис и экосистему PHP приходилось осваивать по ходу.

AI использовался не как «сгенерируй всё за меня», а скорее как наставник: непонятное - спрашивал, отправлял код AI на проверку и правил его логические ошибки, генерировал шаблонный код.

### Как использовался AI

**Обучение и адаптация**
- Освоение Symfony с нуля: «как в Symfony сделать X?»
- Сравнение паттернов: NestJS (декораторы, middleware, pipes) → Symfony (атрибуты, listener'ы, MapRequestPayload)
- Разбор ошибок: «падает с ошибкой X - почему и как исправить?»

**Проектирование и рефакторинг**
- Набросок архитектуры по знакомым-паттернам (Controller → Service → Repository)
- Проверка кода на правильность
- Доработка предложенных фрагментов до рабочего состояния

**Инфраструктура**
- Черновик README и OpenAPI-спецификация сгенерированы AI на основе кода
- Финальные правки и дополнения (примеры ответов, описание архитектуры, блок «Что пришлось исправлять») - сделаны вручную

### Что пришлось исправлять вручную

- **Схема БД:** расхождение `message` / `comment` между миграцией и репозиторием
- **Doctrine DBAL 4:** параметр `url` не работал - перешёл на явный `driver` + `path`
- **Symfony-особенность:** без `Accept: application/json` API отдаёт HTML вместо JSON
- **AI-промпты:** подогнал формулировки под реальное поведение модели
- **Email:** `sleep(10)` из-за rate-limit бесплатного Mailtrap
- **Конфигурация:** подбор рабочих значений `.env` (LM Studio, Mailtrap, rate limit)
- **Документация:** перепроверка README и OpenAPI на соответствие коду
